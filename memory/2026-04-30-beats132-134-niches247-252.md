## 🏹 NICHE #247: GCP Cloud Run Artifact Supply Chain Integrity Validator & Base Image Digest Drift Detector

**B2B Problem:** Platform engineering and security teams deploying services on GCP Cloud Run face a silent supply chain integrity failure — when Cloud Run services are built from GitHub triggers or Dockerfiles with unpinned base images, the resulting container artifact cannot be cryptographically traced to a specific source commit, a verified build environment, or a known-good base image digest. The GCP Cloud Threat Horizons H1-2026 report confirms software supply chain compromises (3% of cases, including compromised NPM packages) as a top threat pattern. Failure modes: (1) a Cloud Run service is built from a GitHub pull request branch instead of a tagged release — the artifact cannot be tied to a specific release SHA, violating SLSA Level 3 provenance requirements, (2) the Dockerfile uses `python:3.11-slim` instead of `python:3.11-slim@sha256:...` — between builds, the base image digest changes (new CVE patched into the tag), and the artifact's provenance chain is broken because the build is no longer reproducible, (3) Cloud Run's SLSA build level compliance panel shows a service at SLSA Level 2 but it was deployed without a valid attestation, meaning the provenance metadata was never validated, (4) a Cloud Run service in a production project uses a shared Cloud Build worker pool cross-project — an attacker with access to the shared pool could inject a malicious build step, and the provenance would still show the build ran in a trusted environment, (5) an attacker compromises a third-party NPM package used in the Cloud Run build — the compromised package is baked into the artifact at build time, but without binary authorization attestations, there is no mechanism to detect or prevent deployment of artifacts built with compromised dependencies. No tool continuously validates Cloud Run artifact supply chain integrity end-to-end, detects base image digest drift between builds, verifies Binary Authorization attestation coverage, or forecasts which services will fail SLSA compliance validation. Teams discover supply chain integrity gaps only during compliance audits — often after months of production services have been deployed with unverifiable provenance.

**Target Market:** Platform engineering and security teams at mid-large SaaS companies on GCP using Cloud Run for production services, organizations with SLSA compliance requirements (Level 3+ artifact provenance), DevSecOps teams managing software supply chain integrity for serverless workloads, and compliance teams responsible for SOC 2, ISO 27001, or GDPR artifact traceability for Cloud Run deployments.

## 🏹 NICHE #248: GitHub Actions OIDC AWS Role Trust Policy Drift Detector & Cross-Account Privilege Escalation Forecaster

**B2B Problem:** Platform and DevOps teams using GitHub Actions with OIDC to assume AWS IAM roles face a silent privilege escalation risk — when GitHub's OIDC provider configuration or the IAM role trust policy drifts from the intended secure state, workflows from arbitrary repositories or branches can assume production AWS roles, bypassing intended access controls. The attack surface is massive: GitHub's OIDC provider is shared across millions of repositories, and misconfigured IAM role trust policies that use `repository` or `workflow` subject claim filters without proper `branch` or `tag` conditions allow any workflow in any repository under an org to assume the role. A 2026 analysis of GitHub Actions OIDC AWS configurations by security researchers found widespread permissive trust policies — the most common misconfiguration is an IAM role trust policy that trusts `repo:my-org/*` (all repositories in the org) instead of `repo:my-org/my-specific-repo` (specific repository), combined with no conditions on `workflow_dispatch` vs. `pull_request` vs. tagged release. Failure modes: (1) an IAM role trust policy trusts `repo:my-org/*` without a `ref` condition — a developer creates a workflow on a feature branch that assume-prod-role, and any push to any branch in any repo under the org triggers the role assumption, (2) the trust policy uses `sub: workflow_dispatch` without also requiring `pull_request_target` restrictions — an attacker who can open a PR against any repo in the org could modify the PR branch workflow to assume elevated roles, (3) the trust policy doesn't include `iat` (issued-at) timestamp validation — a workflow that receives a token could replay it beyond the acceptable time window, (4) the GitHub OIDC thumbprint configured in the IAM identity provider is outdated — if GitHub rotates the thumbprint, the identity provider becomes unusable, but the real risk is that a misconfigured thumbprint can allow an attacker to spoof the GitHub OIDC provider, (5) an IAM role has `AssumeRoleWithWebIdentity` permitted from the GitHub OIDC provider, but the role's permission boundary is too broad — if the role is assumed, the attacker gains full access to all actions permitted by the boundary, including `s3:GetObject` on all production buckets or `lambda:InvokeFunction` on all functions, (6) the `aws:PrincipalTag` tags from the GitHub OIDC token are used in IAM policies for access control, but the tags are not validated against an allowlist — an attacker could inject arbitrary tags into the OIDC token via a malicious workflow, gaining access to resources protected by tag-based policies. No tool continuously monitors IAM role trust policies for OIDC drift, detects when repository or branch conditions have become overly permissive, forecasts which roles are at highest risk of cross-account privilege escalation, or validates that GitHub OIDC thumbprints are current. Teams discover OIDC trust policy misconfigurations only during security audits — often after a breach has already occurred.

**Target Market:** Platform and DevOps teams at mid-large companies using GitHub Actions with OIDC to assume AWS IAM roles, organizations with multiple AWS accounts (dev/staging/prod) where cross-account role assumption is used, security teams responsible for IAM credential lifecycle management, and DevSecOps teams managing GitHub organization-level OIDC integrations with AWS.

**Technical Architecture:**
- Python/Go collector via AWS IAM API (get role, get role policy, list roles, get open-id connect provider info) + AWS CloudTrail API (lookup events for AssumeRoleWithWebIdentity, GetFederation, WebIdentity) + GitHub API (list organization workflows, get repository OIDC policy configuration, list organization OIDC custom roles) + GitHub OIDC discovery endpoint (fetch current thumbprints, issuer configuration) + AWS Security Hub (findings for IAM-related controls) + AWS Config (recorder for IAM policy changes)
- IAM role trust policy drift detector: for each IAM role with a trust policy that includes `oidc.eks.amazonaws.com` or `token.actions.githubusercontent.com`, parses the condition keys — extracts `repo`, `workflow`, `environment`, `ref` conditions — compares against the intended policy state stored in GitOps (Terraform, CloudFormation) — alerts when the live role's trust policy is more permissive than the intended state (e.g., `repo:my-org/*` instead of `repo:my-org/my-repo`)

- OIDC thumbprint currency validator: fetches the current GitHub OIDC thumbprints from `https://token.actions.githubusercontent.com/.well-known/openid-configuration` — compares against the thumbprints configured in each IAM identity provider — alerts when configured thumbprints are outdated or missing current ones

## 🏹 NICHE #249: Kafka Consumer Group Lag Accumulation & Dead Letter Queue Cascade Forecaster

**B2B Problem:** Platform and streaming infrastructure teams running Kafka face a silent data pipeline failure mode — when a consumer group falls behind due to a slow consumer, a deploy, or a processing error, the lag accumulates silently while producers keep writing. Messages in the dead letter queue (DLQ) pile up without alerting, and when the DLQ fills or the consumer eventually fails, the accumulated unprocessed messages create a cascade of downstream data gaps, reordered events, or duplicate processing. The failure pattern is insidious: lag builds up over hours or days, the DLQ grows unbounded, and by the time the team notices, hundreds of thousands of messages need replay or the downstream service has already served stale/hole-filled data to users. Failure modes: (1) a consumer group falls behind after a code deploy that introduced a processing regression — the lag grows from 1,000 to 10 million messages over 48 hours, and by the time alerting triggers, the team must choose between a dangerous catch-up replay or accepting data gaps in the time window, (2) a Kafka topic's DLQ is configured with a retention policy that expires messages before the DLQ consumer processes them — messages are permanently lost with no replay capability, (3) a consumer group is scaled down from 6 to 2 partitions after a cost-cutting exercise — lag instantly explodes as the remaining consumers can't keep up with the producer rate, but no alert fires because the partition count change wasn't tracked, (4) a schema registry change to an Avro/Protobuf topic makes old messages unreadable by the new consumer schema — the consumer silently skips batches without marking them as processed, creating invisible data gaps in the output topic, (5) a consumer offset commit failure causes the consumer to re-read the same messages repeatedly — the consumer appears active but is stuck in a re-processing loop, consuming resources without making progress. No tool continuously forecasts consumer group lag growth rate, models when lag will exceed critical thresholds, tracks DLQ depth relative to message expiry policies, or detects when partition count changes will impact consumer throughput. Teams discover lag accumulation only when a business report shows missing data.

**Target Market:** Platform engineering and streaming infrastructure teams at mid-large companies running Kafka for event-driven microservices, data engineering teams responsible for real-time data pipelines on Kafka, organizations with strict SLA requirements on event processing latency, and DevOps/SRE teams managing Kafka clusters with multiple consumer groups across teams.

**Technical Architecture:**
- Python/Go collector via Kafka AdminClient API (list consumer groups, describe consumer group, list offsets) + Kafka Consumer Group API (get lag per topic/partition) + Kafka Topic API (describe configs, get partition count) + Confluent Schema Registry API (list subjects, get schema, list schema versions) + Dead Letter Queue topic metadata (describe topic for DLQ topics) + AWS MSK / Confluent Cloud API (cluster metrics, consumer group metrics) + Prometheus JMX Exporter metrics (consumer lag metrics, poll rate, commit rate)
- Consumer group lag growth forecaster: for each consumer group, samples lag per partition over time — computes lag growth rate (messages/second) — forecasts when lag will exceed a configurable threshold (e.g., 1M messages, 1 hour of data) — alerts before the threshold is crossed so the team can investigate or scale before data loss risk
- DLQ depth vs. message expiry risk model: identifies DLQ topics associated with each consumer group — tracks DLQ depth (message count, byte size) — fetches the DLQ topic's retention policy — computes the time window before messages in the DLQ expire before being consumed — alerts when DLQ depth indicates messages will expire before processing capacity catches up
- Partition count change impact analyzer: monitors partition count for all topics with active consumer groups — when partition count changes (increase or decrease), recalculates the effective throughput capacity of the consumer group — alerts when a partition change will cause lag to exceed thresholds (e.g., adding partitions without adding consumers creates hot-partition risk; removing partitions without rebalancing creates immediate lag spike)
- Schema evolution gap detector: for each topic, fetches the current schema from the schema registry — compares against historical schema versions used in the consumer group — detects when a schema change would make old consumer group offsets unreadable — alerts when a schema change is detected but the consumer group hasn't been migrated to a compatible schema version
- Re-processing loop detector: monitors consumer group offset commits vs. messages consumed — flags when the consumer is processing messages but the committed offset is not advancing (same offset range being reprocessed repeatedly) — distinguishes between normal rebalancing and a stuck consumer in a re-processing loop consuming resources without making progress
- Kafka AdminClient API + MSK/Confluent Cloud API + Schema Registry API + Prometheus JMX metrics → PostgreSQL + Grafana: consumer group lag growth timeline per group/topic, DLQ depth vs. message expiry risk heatmap, partition count change event log with impact assessment, schema evolution compatibility matrix, re-processing loop detection alerts, lag forecast threshold crossings

## 🏹 NICHE #252: Ephemeral Environment Lifecycle Orphan Detector & Multi-Cloud Cleanup Sentinel

**B2B Problem:** Platform and DevOps teams using ephemeral environments (review apps, preview environments, branch deployments) face silent environment orphaning — when a PR is closed or merged, the cleanup pipeline should destroy the associated ephemeral environment, but when cleanup fails silently, cloud resources (Kubernetes namespaces, AWS RDS instances, S3 buckets, GCP Cloud Run services) persist indefinitely, accumulating cost and expanding attack surface. At scale (50+ PRs/day), even a 1% cleanup failure rate creates 10+ orphaned environments per week. Failure modes: (1) a developer merges a PR and the cleanup Kubernetes job runs but fails due to a permissions issue — the namespace `pr-4521-payment-svc` persists with live cloud resources costing $50/day, (2) the ephemeral environment's Kubernetes namespace is deleted but the AWS RDS instance and S3 bucket attached to it survive — the database and bucket remain active with prod data snapshots accessible to anyone with the AWS account, (3) a security team runs vulnerability scanning but never scans ephemeral environments because they don't know they exist — CVE-2024-5678 affects 12 orphaned environments running outdated nginx versions, (4) a compliance auditor finds 200 active cloud resources with no corresponding service catalog entry — the environments aren't in any CMDB, violating SOX/ISO27001 asset inventory requirements, (5) an ephemeral environment's DNS record (`pr-4521.app.acme.dev`) persists after namespace deletion — an attacker who registers the abandoned subdomain gets a valid TLS certificate for a domain that appears to belong to acme.com. No tool continuously monitors ephemeral environment lifecycle state across all cloud providers and CI/CD systems, detects orphaned resources from failed cleanup jobs, forecasts environment expiry based on PR state, or auto-generates cleanup remediation. Teams discover orphans only during monthly cost reviews or security audits.

**Target Market:** Platform engineering and DevOps teams at mid-large SaaS companies with ephemeral environment pipelines (GitHub Actions, GitLab CI, Argo Workflows), organizations with 20+ ephemeral environments provisioned daily across Kubernetes + AWS + GCP, security teams responsible for attack surface reduction, and finance teams managing cloud spend across development accounts.

**Technical Architecture:**
